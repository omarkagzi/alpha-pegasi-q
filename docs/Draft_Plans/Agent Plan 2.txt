What an LLM Policy Router looks like in practice

Right now the repo has a provider abstraction, but it is only half the solution. createProvider() supports gemini, openrouter, and groq, while deepseek currently throws an error. The human chat route already selects provider credentials from the agent record and environment, but heartbeat/journal/post-processing are still not fully governed by one central policy layer.

The router should be a single decision point that answers:

“Given this feature, this tier, this user’s remaining credits, and current provider health, which model do we use, with what limits?”

In practice:

type Feature =
  | 'chat'
  | 'heartbeat'
  | 'journal'
  | 'summary'
  | 'sentiment';

type Tier = 'visitor' | 'explorer' | 'creator' | 'studio';

interface PolicyContext {
  tier: Tier;
  creditsRemaining: number;
  activeAgents: number;
  byoKeyEnabled: boolean;
  providerHealth: Record<string, 'ok' | 'degraded' | 'down'>;
}

interface LlmPolicy {
  provider: 'groq' | 'gemini' | 'openrouter';
  model: string;
  maxTokens: number;
  temperature: number;
  responseFormat: 'text' | 'json';
  fallbackProvider?: 'groq' | 'gemini' | 'openrouter';
}

function choosePolicy(feature: Feature, ctx: PolicyContext): LlmPolicy {
  // 1. enforce tier caps
  // 2. pick cheapest acceptable provider
  // 3. pick feature-specific model
  // 4. fall back only to allowed providers
  // 5. log usage to ledger
}

The important part is not the function shape; it is the behavior:

Enforce quota before the request runs.
Choose provider/model from policy, not from scattered route logic.
Send the request through one adapter path.
Write usage back to the ledger.
If a provider is unsupported or down, fall back to a tier-approved provider instead of crashing.

That makes “swap the brain without losing continuity” real, because the world logic does not care whether the active model is Groq today, OpenRouter tomorrow, or a future adapter later. Right now the repo is close, but not there yet, because deepseek is a stub and provider choice is still partly feature-local.

What I meant by the “credit tables”

I was referring to the tables already in your schema: users.world_credits_balance and world_credit_ledger. Those are the monetization primitives I meant. There is no subscription billing table in the initial schema snapshot I reviewed, and I do not see a Stripe dependency in package.json, so today the repo has usage accounting, not payments infrastructure.

The cleanest path is:

keep world_credits_balance as the included monthly usage balance

That way you are not mixing money, entitlement, and simulation state into one column.

One thing I would change in your plan: do not anchor monetization on “BYO key” too early. Treat BYO key as a Studio-tier advanced feature after the quota system is in place, not as a core launch assumption.
