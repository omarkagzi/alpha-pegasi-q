A

Route-Based Separation (Recommended)

Three.js lives at /world. Phaser lives at /world/arboria and /world/arboria/[settlement]. Each route mounts its own renderer. No shared wrapper needed — Next.js routing handles the switch. Phaser is dynamically imported and initialized when the Arboria route mounts, destroyed on unmount.

Pros

  • Cleanest separation of concerns
  • No complex state orchestration
  • Each view manages its own lifecycle
  • Works naturally with Next.js App Router

Cons

  • ~300-500ms Phaser cold start on first navigation
  • Full page transition (not seamless fade)
  • Phaser state lost if user navigates away and back
B

WorldCanvas Wrapper (PRD Spec)

A parent <WorldCanvas> component at the /world layout level pre-initializes both Three.js and Phaser on load. Both canvases exist simultaneously — one hidden, one visible. Fade transitions between them. Neither is destroyed.

Pros

  • Zero transition lag (both pre-loaded)
  • Seamless visual transitions
  • Matches PRD §8.2 exactly
  • Phaser state preserved across navigation

Cons

  • Higher initial memory footprint
  • More complex lifecycle management
  • Must coordinate two game loops
  • Harder to debug

My recommendation: Option A (route-based) for MVP. The PRD specifies the WorldCanvas wrapper, but that's an optimization. For MVP with just Arboria, the cold-start delay is acceptable and the simpler architecture will save significant development time. We can refactor to the WorldCanvas wrapper in a later phase when more biomes are realized and instant transitions matter.
