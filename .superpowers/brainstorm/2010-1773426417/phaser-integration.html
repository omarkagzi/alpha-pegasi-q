A

Route-Based Separation (Recommended for MVP)

Three.js lives at /world. Phaser lives at /world/arboria and /world/arboria/[settlement]. Each route mounts its own renderer. No shared wrapper — Next.js routing handles the switch. Phaser is dynamically imported when the route mounts.

Pros

  • Cleanest separation of concerns
  • No complex state orchestration
  • Each view manages its own lifecycle
  • Works naturally with Next.js App Router

Cons

  • ~300-500ms Phaser cold start on first nav
  • Full page transition (not seamless fade)
  • Phaser state lost on navigate away
B

WorldCanvas Wrapper (PRD §8.2 Spec)

A parent <WorldCanvas> at the /world layout level pre-initializes both Three.js and Phaser on load. Both canvases exist simultaneously — one hidden, one visible. Fade transitions between them. Neither destroyed.

Pros

  • Zero transition lag (both pre-loaded)
  • Seamless visual transitions
  • Matches PRD §8.2 exactly
  • Phaser state preserved across nav

Cons

  • Higher initial memory footprint
  • More complex lifecycle management
  • Must coordinate two game loops
  • Harder to debug

My take: Option A for MVP. The PRD's WorldCanvas wrapper is the right long-term architecture, but for MVP with only Arboria, the 300ms cold-start is acceptable and the simpler code will save significant dev time. We can refactor to the wrapper later when more biomes need instant transitions.
